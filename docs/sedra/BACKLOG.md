### Fix the hint bar's 'Shift+drag pan' lie
[canvas-experience · S · impact 3] Every hint string ends with 'Shift+drag pan · wheel zoom' (editor.ts:1809-1811) but shift+left-drag was repurposed for additive selection (comment at editor.ts:2110-2112) — pan is middle/right-drag only, so new users follow the one pan affordance offered and get a marquee. Change the suffix to 'Middle/right-drag pan · wheel zoom' and only append it to tool-level hints, not transient operation hints. Cheapest credibility fix on the canvas.

### Hover pre-selection feedback on canvas (wire up the dead .hover-box)
[polish-details + canvas-experience · M · impact 5] Merged duplicate findings — both lenses called this the single biggest 'demo vs pro' tell. Nothing telegraphs what a click will hit, doubly bad since wire selection is segment-level. The infrastructure half-exists: DrawPartOpts.hover and .hover-box CSS (glyphs.ts:965-975, index.html:434-437) are defined but never used, and mousemove already renders every move. In the select-tool fallback of the mousemove handler (editor.ts:~2187) run pickAt(world), stash state.hoverTarget, re-render only on change; in render() pass hover:true to drawPart for the hovered part and draw a wider translucent --select stroke over the hovered wire segment (skip while wireDraft/moveDraft/boxSelect are live); toggle a cursor class on #canvas-wrap; apply the same treatment in delete/rotate/highlight tools so the user sees exactly what the click will consume.

### Double-click / F2 inline value editing
[power-ergonomics + canvas-experience · M · impact 5] Merged duplicate findings. Editing values is THE core loop of schematic capture, yet dblclick on a part is a dead gesture (handler at editor.ts:2610 returns early unless the wire tool is active) and F2 is unbound. When tool is 'select' and pickAt hits a part: select it and spawn a small floating <input> positioned over the part's value label, pre-filled with p.value, commit on Enter/blur, cancel on Esc, reusing the commit logic from refreshProps' Value field (editor.ts:2796). Bind F2 (and Enter with one part selected) to the same editor; dblclick on a wire opens the net-label editor. Cheap fallback: dblclick/F2 selects the part and focuses+selects the props-pane Value input.

### Ctrl+D instant duplicate
[power-ergonomics · S · impact 4] Duplicating currently takes four interactions (Ctrl+C, anchor click, Ctrl+V, place click); Ctrl+D currently falls through to the browser bookmark dialog (return at editor.ts:3474). In the ctrl/meta keydown branch add k==='d': snapshot the selection (same extraction as finalizeCopyAnchor, editor.ts:3676) using the bbox corner as implicit anchor, insert clones offset one GRID step down-right, select them, and enter startMove(..., freshlyPasted=true) — the existing paste hand-off at editor.ts:3770 — so the user confirms in place or slides them. preventDefault to beat the browser.

### Arrow-key nudge of the selection
[power-ergonomics + feature-gaps · S · impact 4] Merged duplicate findings — no arrow handling exists at all, and the connectivity-preserving move machinery already does the hard work. With a non-empty selection, no focused input, and no active draft: arrows translate the selection one GRID step (Shift+arrow = 4) via the startMove → immediate-commit path so junction healing, parity check and history all apply. Coalesce nudge bursts (~400-500ms debounce) into one undo entry. With nothing selected, arrows pan the canvas ~80px for keyboard-only navigation.

### '?' shortcut cheat-sheet overlay
[power-ergonomics + information-architecture · S · impact 4] Merged duplicate findings. SEDRA's rich shortcut set (r/l/c/v/i/d/w/g/s/x/b/h, m, u, f, Space, Esc, Del, Ctrl combos) is only discoverable one fragment at a time via the rotating hint bar. Bind '?' to a centered overlay listing all bindings in columns — Place / Edit / View+Analysis — using the existing kbd styling; dismiss on Esc/'?'/click-away. Generate the rows from the shortcut registry created in the tooltip fix so cheat sheet, tooltips, and keydown map can never drift. The cheapest possible 'this is professional software' signal.

### Trackpad-correct wheel handling: pinch/Ctrl zooms, two-finger scroll pans
[canvas-experience · S · impact 4] The wheel handler (editor.ts:2301-2314) treats every event as zoom from deltaY and discards deltaX and e.ctrlKey — on laptops (a primary device for a browser tool) two-finger pan zooms, horizontal motion does nothing, and OS pinch is indistinguishable from scroll. Robust rule: e.ctrlKey → zoom anchored at cursor (covers pinch); deltaX !== 0 → pan both axes (state.pan -= delta); otherwise keep zoom so mouse wheels behave as today. Update the hint text accordingly.
